§5 技术债清单

来源：代码审查 + T-001/T-002 交付报告。TD-1～TD-7 已由 T-004 清偿；TD-8 保留（正确设计）。

ID描述影响级别来源
TD-1 向导路由守卫漏洞：router.ts 中 wizard 守卫仅在导航到 /dashboard 时触发。向导已处理后直接访问 /wizard 不被重定向，用户可重复进入向导流程 中 T-002 delivery 已知问题 已修复（T-004）
TD-2 IP 解析重复代码：downloader 包已导出公网 IP 解析函数，但 handlers_system.go 的 fetchIPFromURL 有自己的内联 JSON 解析，未复用 低 T-002 delivery 已知问题 已修复（T-004）
TD-3 verify_all 前端检查永久 SKIP：verify_all.sh 在项目根查找 package.json，但 package.json 在 web/ 目录，导致 B.1 typecheck、B.2 lint、B.3 unit tests 全部 SKIP，前端质量门禁实际上未执行 中 代码审查 + scripts/verification_history.log 已修复（T-004）
TD-4 Version 字符串写死：var Version = "0.1.0" — 若构建时未注入 -ldflags "-X main.Version=..." 则版本永远为 0.1.0，无法区分构建 低 cmd/frp-easy/main.go 已修复（T-004）
TD-5 slog 单写模式：newLogger 在 logFile 非 nil 时仅写文件，stderr 不收 slog 输出（仅 startup banner 写 stderr）。go run 开发模式下 slog 日志写入文件，不便观察 低 cmd/frp-easy/main.go:newLogger 已修复（T-004）
TD-6 dist/ .gitignore 歧义：根目录 .gitignore 的 dist/ 规则递归匹配，会忽略 internal/assets/dist/（Go embed 的输入目录）。克隆后须先 npm run build 才能 go build；此依赖关系已在 README "前置条件"和"更新流程"中文档化。根本修复策略（是否提交 dist/ 到 git）见 OPT-3 中 build.sh + .gitignore + internal/assets/assets.go 已修复（T-004，文档化方式 OPT-3 选 B）
TD-7 auto-restore 时 TOML 缺失无预检：autoRestoreProcs 在 frpc.toml / frps.toml 不存在时直接 Start，子进程立即失败，状态标记为 error，UI 无法区分"配置未生成"与"进程崩溃" 低 cmd/frp-easy/main.go:autoRestoreProcs 已修复（T-004）
TD-8 单 SQLite 连接并发限制：SetMaxOpenConns(1) 使所有读写串行化，适合 MVP（≤200 proxy）但在多 tab 高频轮询时（logs/proc/status 均 2s 轮询）会产生队列等待 低 internal/storage/store.go

T-011 网络默认值变更的取舍说明

T-011 把 UI 默认绑定地址从 127.0.0.1 改为 0.0.0.0，是一次有意识的取舍，理由与配套保障如下：

维度说明
为何改frp_easy 本质是远程内网穿透管理工具，运维场景天然需要从其他设备访问 Web UI；旧默认 127.0.0.1 迫使每个用户首启后手动改配置
认证加固0.0.0.0 不等于"无认证暴露"——项目已内置 argon2id 密码哈希、会话 Cookie（HttpOnly + SameSite）、CSRF 防护、登录失败 IP 限流
暴露窗口最危险的是 setup 完成前（界面尚无管理员账号）；启动期 stderr 会打印中性安全提示，引导用户尽快完成 setup
如何改回仅需本机访问的用户，把 frp_easy.toml 的 UIBindAddr 改为 127.0.0.1 后重启即可；老用户显式写过的配置不会被升级静默改写

§8 更新流程说明

警告：仅 git pull + 重启旧二进制是不够的 二进制不变则更新不生效；跳过 npm run build 则前端仍为旧版本。

完整更新步骤（Linux / macOS）

git pull
bash scripts/build.sh
# 以新二进制替换旧进程并重启

完整更新步骤（Windows）

git pull
.\scripts\build.ps1
# 以新二进制替换旧进程并重启

原因说明

#原因技术细节
1 前端需重建 internal/assets/dist/ 被 .gitignore 的 dist/ 规则排除，git pull 不会更新前端产物。前端 SPA 通过 //go:embed all:dist 嵌入 Go 二进制，build.sh 自动执行 npm run build
2 数据库迁移自动运行 storage.Open() 在启动时对所有未应用的迁移执行 applyOne()，用户无需手动执行任何 SQL
3 配置文件向后兼容 appconf.Load() 对现有 frp_easy.toml 缺失字段补默认值；frp_easy.toml 在 .gitignore 中，git pull 不覆盖用户配置
4 仅后端变更的简化路径 若明确只有 Go 代码变更，可执行 CGO_ENABLED=0 go build ./cmd/frp-easy 后重启。不确定时始终使用 scripts/build.sh
